Easy-to-manage, automated data protection

The Product

In 2011 Seagate Technology make a big push for extending their cloud disaster recovery services. Their goal was to create a great value service that customers can use with minimal technical support.

Seagate EVault Web Access, a web application for centralized management of backed up data on the cloud was created.

The Job

In the summer of 2012 I was hired as a full time web developer and a UX designer to lead and oversee the user experience of the application.

My objective is to advocate for the users by utilizing user-centered design methodologies when designing new or improving existing features for the product. My daily tasks include creating, maintaining and enforcing UI standards and mentoring the rest of the team on usability principles and user experience.

My personal goal is to have everyone on the team understand that the user experience is everyone's responsibility.

Seagate EVault Web Access - UI Refactoring

The Problem

The brand new EVault Web Access application was intended to replace a few other products. As a consequence, it had to do everything all the other applications did.

The team originally tasked with writing it was mainly consisted of software engineers whose expertise was software architecture, not UI design. As a result, very little effort was put towards creating a user-centric application. The UI and workflows were copied with almost no modifications from close to a decade old products intended to be retired. Many workflows were redundant or overly complex, and the UI felt dated even before the application was even released.

A first public release was 5 months away, and with the UI in the current shape, the product was facing a risk of being shut down.

My Role

My first task was driving the UI redesign and optimization of existing workflows. I did competitive and functional analysis of the application, outlined problematic workflows and prioritized their refactoring order based on severity. I redesigned the UI by creating high fidelity prototypes that I used to to convey my vision to the product stakeholders. I finally refactored the code.

User Insights

I communicated with internal users to understand expectations and identify pressure points.

Competitive Analysis

I analyzed how similar tasks are performed in competing products and compared with Web Access.

Product Vision

I created interactive wireframes and prototypes to describe the product vision, usability principles and content strategy.

Coordination

I worked with a team of developers and quality assurance analysts to make sure that the redesign has a minimal impact on ongoing development effort and product launch date isn't impacted.

UI Design, Information Architecture & Validation

I identified and optimized sub-optimal workflows and designed more functional layouts, controls and validation. I collected and analyzed feedback from users.

UI Implementation

I re-wrote the entire UI layer, created reusable HTML templates and refactored and normalized scripts and stylesheets.

Discovery

I conducted user interviews with users who use the the products to be replaced, as well as competitor products.

The most overwhelming response I got was that the users don't like to spend any more time using the application than they absolutely have to and that they don't want to be looking for information, but expect to find it in logical places. This suggested that workflow optimization and contextualization had to be my first priority. I had to eliminate all redundant clicks and show as much relevant information at once.

When asked how much they cared about the attractiveness of the application, to my surprise, 3 in 4 people said they didn't very much and only 1 said that they'd prefer a more attractive application.

So I did a visual exercise with the same users, showing them screen shots of 2 identical applications - the original Web Access application and my prototypes, which I considered visually superior. The numbers flipped - 3 in 4 people said that they prefer the prototype and only 1 was indifferent.

It was clear that our users valued aesthetics even though they weren't specifically asking for it. But the aesthetics could not be getting in the way of the tasks the users were trying to achieve.

When asked about things they would improve in competitor's products, a common suggestion was to create bulk actions and ability to automate tasks. This confirmed the findings from my competitive analysis.

Vision

My vision for Web Access was to be the best backup process management tool on the market, and not just another mediocre tool.

The goal was not to offer an exhaustive amount of features that may or may not be useful, but offer features that the users will actually use, and create them in a way that the users will want to use them.

Design Process

To achieve the vision, I had to address the user's concerns before anything else. That meant simplifying the workflows by providing context and showing more relevant information.

Designing for Scalability

One problem with the existing UI was the disregard for screen real estate. As this application was only intended to run in a web browser on a desktop computer, preserving screen real estate was not taken into consideration. The low scalability of the UI became apparent even more as the application functionality grew. One screen just wasn't going to be enough.

I designed scalable and reusable controls such as drop-down menus, date and time pickers and popup windows. I also created responsive layouts that would scale the UI for different devices. This way more of the screen could be used for information and less for action items.

Designing new features

One of the features that got copied over from the old products was the reporting mechanism. The reports consisted of large quantities of data just displayed to the user in a tabular format with little-to-no interactivity. Data could be sorted but not searched, and the data would start to losing meaning as the amount of data increased.

Reporting was going to be one of the most frequently used sections of the application as users could directly link the reported data to their billing.

In the user interviews I learned that the users would appreciate a way to filter and sort their data. They also mentioned that a graphical representation of the data may be useful if done right.

I was debating between replacing the tabular data reporting with a graphical reporting and improving the tabular and adding graphical reporting. This was also a concern due to the approaching release date.

I created low fidelity functional mockups to show the interactivity to the involved parties after what I was given a green light to proceed with the design - this was one feature everyone fell in love with.

Optimizing and improving workflows

A user described to me the steps they take when troubleshooting an error in their system. What caught my attention was how they go about viewing log files. The original log viewer in the application was essentially displaying a text file in a browser with no extra functionality. So the user had to go through the contents, find the error codes, copy them into Notepad, open the knowledge base website and search for information on the error code, often times not finding any results. Ideally never, but in the real world, a user would have to go through this painful process a few times every week.

My goal was to make the log viewer the last place a user searches for information, and do it in a way that will require minimal effort.

The mission was accomplished. The new log viewer became one of the most loved features of Web Access.

Challenges

The biggest challenge I faced was balancing the expectations of different layers of stakeholders. At every design iteration the feedback received from all involved parties would range from slightly contradicting to completely opposite.

For example, many of the users preferred context driven menus because it would speed up navigation, but the product owners preferred static menus because it improves the discoverability of the application.

The solution was often a compromise, in this case, I created static menus in which non-applicable items are disabled and pushed to the bottom.

Validation

Ever since the initial launch, Seagate EVault Web Access has been greatly accepted by the users. Steep conversion rate allowed the company to retire the old products Web Access was intended to replace ahead of schedule, with the last one to be retired by 2016, with what Web Access becomes Seagate's only web application for managing cloud backup processes, being used by thousands users on a daily basis.

I hold user interviews after every release to get more insight into what the users need, what is causing them pain and what they love about the application.

At the moment I am working on redesigning the workflows revolving around filling out complex forms to use wizard like navigation how the users can remain focused on the specific information they are providing without any distractions.

Lessons Learned

Users aren't as excited about my application as I am

As exciting as that popup transition may seem, users will likely trade it for spending less time using the application in a heartbeat. In the eyes of the user, optimized workflows are infinitely more useful than pretty interfaces. With that in mind, 3 of 4 people choose the prettier interface, and the 4th one may or may not choose it. But nobody goes for the ugly one on purpose.

Users don't know what they want

... but they know what they don't want.

So eliminate what they know they don't want, but don't be afraid to offer something different from what they are used to.
