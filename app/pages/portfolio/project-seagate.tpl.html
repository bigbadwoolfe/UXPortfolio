Easy-to-manage, automated data protection

Product

In 2011 Seagate Technology make a big push for extending their cloud disaster recovery services. Their goal was to create a great value service that customers can use with minimal technical support.

Seagate EVault Web Access, a web application for centralized management of backed up data on the cloud was created.

The Job

In the summer of 2012 I was hired as a full time a UX designer and web developer and to lead the user experience of the application.

My objective was to advocate for the users by utilizing user-centered design methodologies when designing new or improving existing features for the product. My daily tasks included user analysis, requirements specification, application design and usability testing. I was also driving the creation, maintenance and enforcement of UI standards for the application and mentoring the rest of the team on usability principles and user experience.

I set a personal goal to make sure everyone on my team understand that design and user experience is everyone's responsibility.

"Design is not just what it looks like and feels like.
Design is how it works"
Steve Jobs

Vision

My vision for Web Access was to be the best backup process management tool, and not just another mediocre web application on the market that users.

The company's goal was not to offer an exhaustive amount of random features, but offer features that would solve actual problems for customers, and have them created in a way that they will become customers' choice.

This perfectly aligned with vision for Web Access.

Seagate EVault Web Access - UI Refactoring

The Problem

The brand new EVault Web Access application was intended to replace a few other products. As a consequence, it had to do everything all the other applications did.

The team originally tasked with writing it was mainly consisted of software engineers whose expertise was server-side software architecture, not UI design. As a result, no effort was put towards creating a user-centric application. The UI and workflows were a replica of the ones found in the almost a decade old applications scheduled for retirement. Most workflows were either redundant or overly complex, and the UI was sloppy and inconsistent throughout the application and felt dated even before the application's go-live date scheduled for 5 months later.

"You never get a second chance to make a first impression."

My Role

My first task was driving the UI redesign and optimization of existing workflows. I had to stop the bleeding before it caused any more damage.

I did functional and competitive analysis of the application, outlined problematic workflows and prioritized their refactoring order based on severity. I redesigned the UI by creating high fidelity prototypes that I used to to convey my vision. I explained and defended my design decisions to the product stakeholders. Upon design approval by the company, I lead the UI refactoring and coordinated the validation.

Competitive Analysis

I familiarized myself with competitor products analyzed how similar tasks are performed.

User Insights

I conducted systematic interviews to understand expectations and identify pain points users have when using the existing in-house and similar competitor products.

Product Vision

I created interactive wireframes and prototypes to describe the product vision and content strategy.

Coordination

I worked with a team of developers and quality assurance analysts to make sure that the redesign has a minimal impact on ongoing development efforts and product launch date isn't affected.

UI Design, Information Architecture & Validation

I identified and optimized sub-optimal workflows and designed more functional layouts, controls and validation. I created precise UI specifications and conducted user testing and analyzed results, as well as collected user feedback.

UI Implementation

I re-wrote the entire UI layer, created reusable HTML templates and refactored and normalized scripts and stylesheets.

Discovery

I prepared questionnaires and conducted user interviews with target users - customers using the already existing in-house products, as well as technical support analysts familiar with both internal and competitor products.

The most overwhelming response I got was that the users don't like to spend any more time using the application than they absolutely have to and that they don't want to be looking for information, but expect to find it in logical places. This suggested that workflow optimization and contextualization had to be my top priority. I had to eliminate all redundancy and show as much relevant information at once.

When asked how important the attractiveness of the application is on a scale from 1 to 10, to my surprise, approximately 75% of the interviewees answered 4 or below and only about 15% valued attractive UIs at 7 or above.

So I did a visual exercise with the same users, showing them screen shots of 2 applications that serve identical purpose - the original Web Access and my high fidelity prototypes, which I considered visually superior - it was precise, consistent, symmetrical and less cluttered. The result - around 80% picked the prototype as their choice and the rest said could go either way. Nobody was adamant about choosing the inferior UI.

It was clear that our users valued aesthetics even though they weren't specifically asking for it. But the aesthetics could not be getting in the way of the tasks the users were trying to achieve.

When asked about things they would improve in competitor's products, a common suggestion was to create bulk actions and ability to automate tasks. This confirmed the findings from my competitive analysis.

Although 75% of users stated they didn't care about how visually pleasing the user interface was,
80% went for the superior one when asked to choose between two.

Design

User interviews delivered a clear message - workflows had to be simplified and more information had to find it's way in relevant locations. Additionally, users responded positively to the enhanced UI, so a visual face-lift was imminent.

Designing for Scalability

One problem with the existing UI was the disregard for screen real estate. As this application was originally only intended to run in a web browser on a desktop computer, preserving screen real estate and responsiveness were never taken into consideration. The poor scalability of the UI became apparent even more so as the application functionality grew, particularly on lower screen resolutions.

I designed scalable and reusable controls such as drop-down menus, date and time pickers and popup windows. I also created responsive layouts that would scale the UI for different devices. This way more of the screen could be used for information and less for action items.

Designing new features

One of the features that got copied over from the old products was the reporting mechanism. The reports consisted of large quantities of data displayed to the user in a tabular format with almost no interactivity. Data could be sorted but not searched, and the data would start losing meaning as the amount of information shown increased.

Reporting was going to be one of the most frequently used sections of the application as users could directly link the reported data to their billing.

In the user interviews I learned that the users would appreciate a way to filter and sort their data.

Even with improved searching and sorting, large amounts of data in a table are likely to provide an inferior user experience. So I set out to aggregate the data into an interactive graph.

I prototyped how the data would be more effectively shown and created a low fidelity functional mockup to better support design. Despite the quickly approaching launch date, this new unplanned feature was given the green light.

This was one feature everyone fell in love with and users quickly abandoned the tabular reports even though they were refactored to allow for searching and sorting.

Optimizing and improving workflows - the Log Viewer

In feedback collected from a user, a major pain point was revealed. Any time a backup process completes unsuccessfully, an administrator needs to go through the generated log files and see why the process failed. What caught my attention were the steps needed to get the required information beyond checking the log files themselves.

  1. Go through thousands of lines of log files and search for error codes
  2. Write error codes on paper or Notepad
  3. Open Seagate Knowledge Base - external website
  4. Manually search each error code
  5. If any results, open each result in a new window to get the information
  6. If no results, all that time was just wasted - call support

This was a very manual and tedious process, requiring constant switching between multiple application and juggling multiple error codes at once.

My goal was to design a log viewer that would be the last place an administrator goes for information, and do it in a way that will require minimal effort.

I created an option to easily filter out error codes from the rest of the log content. This reduced the number of lines an admin sees by up to 95%. I further added more interactivity to the log viewer - a user could see whether any error code has any search results by simply clicking or hovering over the error code. The search results and their content were available in the same window with another click.

No more searching through thousands of lines of text, switching between application or juggling error codes.

Mission accomplished successfully. The new log viewer became one of the most loved features of Web Access.

Precise design documentation

I documented the design so that developers can recreate and populate layouts easily, ensuring consistent look throughout the application.

Similar documentation for use of typography, images, validation messages and other reusable web components was created.

Design Challenges

The biggest challenge I faced was balancing the expectations of different stakeholder groups . At every design iteration the feedback received from all involved parties would range from slightly contradictory to completely opposite.

For example, many of the users preferred context driven menus because it would speed up navigation. The product owners insisted on static menus because it improves the application discoverability.

The solution was often a compromise. In this case, I created static menus in which non-applicable items are disabled and pushed to the bottom and the relevant items are highlighted and at the top.

Validation

I try to do a small usability study upon every release. I ask users to run through newly created workflows and observe if and where they stumble. I encourage them to think aloud and explain how they make choices about what they do. I also ask for, receive and analyze feedback from customers and document findings.

Ever since the initial launch, Seagate EVault Web Access has been greatly accepted by the users. Steep voluntary conversion rate allowed the company to retire many products ahead of schedule, with the last one scheduled to be retired by 2016. With that, Web Access is set to become Seagate's only web application for managing cloud backup processes.

Lessons Learned

Users aren't as excited about my application as I am

As exciting as that popup transition may seem, users will likely trade it for accomplishing their tasks quicker in a heartbeat. In the eyes of the user, optimized workflows are infinitely more useful than pretty interfaces. With that in mind, 3 of 4 people choose the prettier interface, and the 4th one will likely choose it as well. But nobody goes for the ugly one on purpose.

Users don't know what they want

... but they know what they don't want.

So eliminate what they know they don't want, but don't be afraid to offer something different from what they are used to.

Context switching is expensive

Any time a user needs to switch focus from one task to another, and then come back to the first one, the user experience suffers as does the user's productivity. The same is applicable, when switching between applications.

The job is NOT done

Creating and maintaining good user experience is a continuous and evolving process. On a daily basis I make sure that the new features added fit well with the rest of the product, both in terms of UI and workflows.

I continue to design major features, oversee their development, and gather and analyze feedback.

Seagate EVault Web Access is being used by thousands of users daily who depend on it to manage the backups for hundreds of thousands of workstations and servers. The product is constantly growing and evolving, and while the experience the users have with Web Access allows it to stand with the best in the business, the job is far from done.
