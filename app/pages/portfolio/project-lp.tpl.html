A company on a journey to simplify playing the lottery in groups.

The Ontario Lottery and Gaming Corporation implemented their own version of this application validating the need, but at the same time rendering this one obsolete. Code never left the development servers.

The Service

LotteryPals.com would allow users to buy and track their lottery tickets online, and more importantly, would simplify the process of creating, managing and joining groups.

My Role

I led the design of the application from an idea to a functional product.

I documented user requirements, created use cases and information architecture, designed workflows, graphics, interaction and participated in the actual implementation of the product.

User Research

I communicated with intended users to identify pain points and created a list of features to address the same.

Planning and Project Management

I created the steps that need to be taken to finish the project, from user profiling through architecture and design to implementation. I prioritized and balanced features based on user needs and business goals.

Information Architecture

I produced documents that outline how the information will flow through the application in the most logical way for the user.

UX Design & Validation

I designed the layouts and the UI. I ensured satisfactory results across various screen sizes.

UI Implementation

I wrote the UI with mock data in HTML with CSS and JS, under MVC.NET. I tested for responsiveness, multiple screen sizes and across all popular browsers.

Discovery

The problem at hand had two major sides to consider - how lottery players are organizing themselves into groups and what was the legally correct way to do it. These two were not always the same.

I learned that lottery players who manage groups that play together have the following pains:

  • Collecting money
  • Scanning tickets and distributing copies
  • Skipping draws
  • Distributing small winnings, free plays, etc

The group members and the ones wanting to join a group had a different set of problems:

  • Missing payments (due to vacations)
  • Difficulty to opt out occasionally
  • Finding groups to join

In addition, the Ontario Lottery and Gaming Corp imposes strict rules for playing in groups that a company would be required to follow.

Finally, the company needed modern web presence how they would sign up new players. Here the goal is to explain the business model in simple terms while gaining the customer's trust.

With all this information I could create profiles for each identified user type (public, regular players, group managers, system administrators) and feature sets required by each user type and get a rough idea on how to present this in a self-explanatory way.

Definition

The application is a heavily data-driven one. Different user types would exist, each with different capabilities. For that, I had to create a user-privilege matrix, and once I had that, create the site maps and user workflows.

The two major components would be the public part of the web application and the member's portal.

Design

Due to the nature of the application, it was imperative that users would be left with both a sense of security and playfulness from the very beginning.

"You never get a second chance to make a first impression."

Influencing user feelings

To address the security feeling, I needed to show a professionally designed and built application. Everything had to be air-tight and consistent throughout the application. This includes layout elements, messages and content. There could not be any room for doubt.

To protect the user, possibilities for mistakes were eliminated as much as possible from all workflows. The users were further reassured by confirming actions and they were given the ability to undo any action for a period of time.

To achieve playfulness I used vibrant colors and relevant graphics that pop off the background.

The challenge was to keep a good balance between feeling secure and feeling playful, as these two often do not go hand in hand.

Simplicity

Users were expected to join to alleviate pain. This application mustn't create any new one.

All data entry forms were simplified and heavily contextual. Logical ranges and visual elements to aid the information exchange were created.

Lessons Learned

Don't guess what the users want

... ask them!

User interviews are invaluable tool. They provide validation to some ideas and completely shatter others. It is those users that will drive the business however, so when an idea is shattered, pick up the pieces and make it better.

Stop planning and start doing

No amount of pre-planning will yield a perfect solution the first time. Refining the product through iteration will happen no matter what, so the sooner it starts, the better. Get a good understanding of what is to be built, and start building it.
